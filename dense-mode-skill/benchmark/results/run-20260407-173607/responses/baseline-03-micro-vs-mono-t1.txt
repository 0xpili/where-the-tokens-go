Almost certainly no — not yet, and possibly never.

**The honest case against microservices for a 5-person startup:**

- You don't have the ops headcount. Microservices trade code complexity for infrastructure complexity: service discovery, distributed tracing, network failures, independent deploy pipelines. That overhead routinely consumes 1-2 full-time engineers just to maintain.
- "Slow" in a Rails monolith is almost never fixed by splitting services. It's usually: missing indexes, N+1 queries, synchronous work that should be async, or missing caching. These are solved in an afternoon, not a 6-month migration.
- Distributed transactions and data consistency become your problem instead of Postgres's.

**What actually works at your scale:**

1. **Profile first.** Run `rack-mini-profiler` or check your APM (Scout, Datadog). Bet on N+1s or missing indexes being the culprit.
2. **Move slow work to background jobs.** Sidekiq + Redis handles a lot of perceived slowness.
3. **Add caching.** Fragment caching, HTTP caching, Redis for hot reads.
4. **Scale the monolith vertically or horizontally.** A few more Heroku dynos or a bigger RDS instance is cheap relative to a rewrite.
5. **Extract only if you have a true boundary.** If one domain (e.g., a report generator, image processing) has wildly different scaling needs, extract *that one thing* as a service — not the whole app.

**The rule of thumb:** Microservices solve an organizational scaling problem (many teams stepping on each other). At 5 people, you don't have that problem. The cure would be worse than the disease.

What does "slow" actually mean in your case — slow pages, slow background jobs, slow deploys? That changes the answer.
